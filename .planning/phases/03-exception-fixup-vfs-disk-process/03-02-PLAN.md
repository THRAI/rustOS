**2. VirtIO MMIO transport** (`kernel/src/drivers/virtio_mmio.rs`):

Implement the VirtIO MMIO register interface for QEMU virt machine. The VirtIO-blk device on QEMU virt is at a known MMIO address (typically `0x10001000` for the first virtio device — verify from QEMU virt machine device tree or use FDT parsing from Phase 1).

Define MMIO register offsets per VirtIO spec 1.2 (MMIO transport):
- `MAGIC_VALUE` (0x000), `VERSION` (0x004), `DEVICE_ID` (0x008), `VENDOR_ID` (0x00C)
- `DEVICE_FEATURES` (0x010), `DRIVER_FEATURES` (0x020), `QUEUE_SEL` (0x030)
- `QUEUE_NUM_MAX` (0x034), `QUEUE_NUM` (0x038), `QUEUE_READY` (0x044)
- `QUEUE_NOTIFY` (0x050), `INTERRUPT_STATUS` (0x060), `INTERRUPT_ACK` (0x064)
- `STATUS` (0x070), `QUEUE_DESC_LOW/HIGH` (0x080/0x084)
- `QUEUE_DRIVER_LOW/HIGH` (0x090/0x094), `QUEUE_DEVICE_LOW/HIGH` (0x0A0/0x0A4)

Provide `read_reg(offset) -> u32` and `write_reg(offset, val)` using volatile MMIO reads/writes. All accesses must use `core::ptr::read_volatile` / `write_volatile`.

**3. VirtIO-blk driver** (`kernel/src/drivers/virtio_blk.rs`):

Implement a minimal VirtIO-blk driver:

a) **Virtqueue**: Single virtqueue (queue 0) with descriptor table, available ring, used ring. Allocate these from the frame allocator (`frame_alloc_sync`). Queue size: use `QUEUE_NUM_MAX` from device, cap at 128.

b) **Descriptor format**: Each I/O request uses 3 descriptors chained:
   - Descriptor 0: `VirtioBlkReqHeader { type_: u32, reserved: u32, sector: u64 }` (read-only)
   - Descriptor 1: Data buffer (read-only for write, write-only for read)
   - Descriptor 2: Status byte (write-only, device writes 0=OK)

c) **Device init sequence** (VirtIO spec §3.1):
   1. Reset device (STATUS = 0)
   2. Set ACKNOWLEDGE (1), then DRIVER (2)
   3. Read/negotiate features (for Phase 3: just accept what device offers, no fancy features needed)
   4. Set FEATURES_OK (8), verify it sticks
   5. Configure virtqueue 0: set QUEUE_SEL=0, read QUEUE_NUM_MAX, set QUEUE_NUM, write descriptor/available/used ring physical addresses
   6. Set QUEUE_READY=1
   7. Set DRIVER_OK (4)

d) **read_sector / write_sector**: Synchronous API:
   ```rust
   pub fn read_sector(sector: u64, buf: &mut [u8; 512]) -> Result<(), ()>
   pub fn write_sector(sector: u64, buf: &[u8; 512]) -> Result<(), ()>
   ```
   Build the 3-descriptor chain, add to available ring, notify device (`QUEUE_NOTIFY = 0`), then poll used ring with adaptive spin+WFI (spin ~1000 iterations, then `wfi` in a loop checking used ring index).

e) **Global driver instance**: Use `spin::Once<VirtioBlk>` for the global driver. Init from `rust_main` after frame allocator init.

f) **FDT-based discovery** (optional but preferred): Scan FDT `/soc/virtio_mmio@...` nodes to find the block device. Alternatively, probe known QEMU virt addresses (`0x10008000` down to `0x10001000`) checking MAGIC_VALUE and DEVICE_ID (type 2 = block).

**4. Wire into main.rs**: After frame allocator init, call `drivers::virtio_blk::init()`. Print confirmation: `kprintln!("[virtio-blk] initialized, capacity = {} sectors", cap)`.

**1. lwext4 integration strategy**:

lwext4 is a C library. For a `no_std` kernel, there are two approaches:
- (A) Build lwext4 from C source via `cc` crate in build.rs and link statically
- (B) Use a Rust ext4 reader crate (e.g., `ext4-rs` or hand-roll minimal ext4 parsing)

Per CONTEXT.md, the decision is lwext4 FFI. Use approach (A):

a) Add lwext4 as a git submodule or vendored source under `third_party/lwext4/`. If submodule management is complex, vendor the minimal source files needed (core ext4 + blockdev interface).

b) Create `kernel/build.rs` that uses the `cc` crate to compile lwext4 C sources for `riscv64gc-unknown-none-elf`. Key files: `ext4.c`, `ext4_balloc.c`, `ext4_bcache.c`, `ext4_bitmap.c`, `ext4_blockdev.c`, `ext4_block_group.c`, `ext4_crc32.c`, `ext4_debug.c`, `ext4_dir.c`, `ext4_dir_idx.c`, `ext4_extent.c`, `ext4_fs.c`, `ext4_hash.c`, `ext4_ialloc.c`, `ext4_inode.c`, `ext4_journal.c`, `ext4_mbr.c`, `ext4_mkfs.c`, `ext4_super.c`, `ext4_trans.c`, `ext4_xattr.c`. Provide a minimal `ext4_config.h` that disables journaling and sets `CONFIG_HAVE_OWN_OFLAGS=1`.

c) If building lwext4 from C proves too complex for the executor (cross-compilation, libc stubs), fall back to approach (B): use a pure-Rust ext4 read-only implementation. The `lwext4-rs` crate or `ext4-view` crate may work. Check crates.io for `no_std`-compatible ext4 crates. The key requirement is: mount ext4, open file, read file, list directory. Write support is nice-to-have for Phase 3.

**IMPORTANT**: Whichever approach you choose, the external interface must be the same: the delegate thread calls synchronous FS functions, and blockdev callbacks go through VirtIO-blk.

**3. Delegate thread** (`kernel/src/fs/delegate.rs`):

The delegate is a single kernel thread (not an async task) that serializes all lwext4 operations. Design per CONTEXT.md:

a) **FsRequest enum**: Variants for each FS operation:
```rust
pub enum FsRequest {
    Open { path: ArrayString<256>, flags: u32, reply: oneshot::Sender<Result<FsFileHandle, Errno>> },
    Read { handle: FsFileHandle, buf_ptr: *mut u8, len: usize, reply: oneshot::Sender<Result<usize, Errno>> },
    Write { handle: FsFileHandle, buf_ptr: *const u8, len: usize, reply: oneshot::Sender<Result<usize, Errno>> },
    Close { handle: FsFileHandle, reply: oneshot::Sender<Result<(), Errno>> },
    Stat { path: ArrayString<256>, reply: oneshot::Sender<Result<FsStat, Errno>> },
    ReadDir { path: ArrayString<256>, reply: oneshot::Sender<Result<Vec<DirEntry>, Errno>> },
    Mount { reply: oneshot::Sender<Result<(), Errno>> },
}
```

b) **Bounded channel**: Implement a simple bounded MPSC channel using `IrqSafeSpinLock<VecDeque<FsRequest>>` + `AtomicUsize` count + `WakerList` for backpressure. Capacity = 256. When full, `send().await` registers a Waker and suspends. When delegate pops a request, it wakes one blocked sender.

Alternatively, use a simpler approach: `spin::Mutex<VecDeque<FsRequest>>` with an `AtomicUsize` for the count and a `WakerSlot` for the delegate to sleep on when empty. The delegate thread spins on the queue (with WFI when empty).

c) **Oneshot reply**: Each request carries a reply channel. Simplest implementation: `Arc<OnceCell<T>>` + `AtomicWaker`. The delegate writes the result, then wakes the requesting future.

d) **Delegate thread spawn**: `delegate_spawn()` creates a dedicated kernel stack (allocate 2 pages via `frame_alloc_sync`), sets up a minimal context, and jumps to the delegate loop. The delegate loop:
```
loop {
    let req = channel.blocking_pop(); // spin + WFI when empty
    match req {
        FsRequest::Open { .. } => { /* call lwext4 */ },
        // ...
    }
}
```

Since this is a kernel thread (not async), it can block freely. It runs on its own stack, not the executor. Use a simple approach: spawn it as a regular function call on a dedicated stack using inline assembly to switch sp, or use the existing executor infrastructure by spawning a future that internally blocks (less clean but simpler).

**Simpler alternative for Phase 3**: Make the delegate a regular async task that uses `yield_now()` between operations. Since lwext4 calls are synchronous and short (VirtIO adaptive polling completes quickly), this may be acceptable. The key invariant is: only ONE task ever calls lwext4 functions (serialization). Mark this as a simplification — the dedicated stack-thread can be added in Phase 4 if needed.

e) **Public async API**: Expose async functions that send requests and await replies:
```rust
pub async fn fs_open(path: &str, flags: u32) -> Result<FsFileHandle, Errno>
pub async fn fs_read(handle: FsFileHandle, buf: &mut [u8]) -> Result<usize, Errno>
pub async fn fs_write(handle: FsFileHandle, buf: &[u8]) -> Result<usize, Errno>
pub async fn fs_close(handle: FsFileHandle) -> Result<(), Errno>
```
